# Internationalization (i18n) — FASE P4

Beija usa [`react-i18next`](https://react.i18next.com/) sobre o
[`i18next`](https://www.i18next.com/) core. Detecção via
`i18next-browser-languagedetector` (localStorage → navigator → htmlTag).

Status atual:
- **pt-BR** — source of truth (linguagem do produto).
- **en** — placeholder traduzido, mesmas chaves. Pronto pra revisão profissional.
- **es** — futuro (estrutura já preparada — basta criar `locales/es/*.json`).

## Estrutura

```
frontend/src/i18n/
├── index.ts              ← init + SUPPORTED_LOCALES + changeLocale()
├── resources.d.ts        ← type augmentation pra t() autocompletar chaves
└── locales/
    ├── pt-BR/
    │   ├── common.json   ← nav, actions globais
    │   ├── auth.json     ← signin + onboarding + reactivation
    │   ├── swipe.json    ← deck, ações, distance, announcements
    │   ├── matches.json  ← lista de matches + tempo relativo
    │   ├── chat.json     ← composer + menu + typing
    │   ├── settings.json ← seções + toggles + language picker
    │   ├── profile.json  ← edit + detail modal
    │   ├── moderation.json ← modal de rejeição + reason labels
    │   └── errors.json   ← ErrorBoundary + network + auth errors
    └── en/  (mesma estrutura)
```

Convenção de chave: `namespace.section.key` em snake_case (ex.:
`swipe.actions.like`, `matches.empty.title`). Interpolação Mustache:
`{{name}}`, `{{count}}`. Plurais: `key_one` + `key_other`.

## Como usar num componente

```tsx
import { useTranslation } from 'react-i18next';

function MyComponent() {
  const { t } = useTranslation('swipe');
  return <button aria-label={t('actions.like')}>♥</button>;
}
```

Para chaves de outros namespaces no mesmo componente:
```tsx
const { t } = useTranslation(['swipe', 'common']);
t('actions.like');           // swipe (default do hook)
t('common:actions.retry');   // common
```

Para uso fora de componentes React (utils, libs):
```ts
import i18n from '../i18n';
i18n.t('swipe:distance.near');
```

## Detecção e persistência do locale

Em ordem:
1. **`localStorage.beija_locale`** — escolha explícita do user via Settings.
2. **`navigator.language`** — preferência do sistema.
3. **`<html lang>`** — fallback.
4. **`pt-BR`** — fallback final.

`SettingsScreen → Idioma` mostra `<select>` com PT-BR / EN. Mudar:
- Persiste em `localStorage` (i18next-browser-languagedetector).
- Persiste em `profiles.locale` (migration `20260525200000`).
- Chama `i18n.changeLanguage(locale)` — toda UI re-renderiza.

Edge functions futuras (`notify_new_message`, `notify_match`, deletion
email) leem `profiles.locale` pra escolher template traduzido.

## Adicionar nova língua

1. Crie `frontend/src/i18n/locales/<bcp47>/*.json` espelhando a estrutura
   de `pt-BR/`. Mesmas chaves; ausência fala fallback para `pt-BR`.
2. Adicione o tag a `SUPPORTED_LOCALES` em `src/i18n/index.ts`.
3. Adicione import + entry no `resources` do mesmo arquivo.
4. Adicione ao `check` constraint do migration `20260525200000_profile_locale.sql`
   (criar migration nova). Exemplo:
   ```sql
   alter table profiles drop constraint profiles_locale_check;
   alter table profiles add constraint profiles_locale_check
     check (locale in ('pt-BR', 'en', 'es-ES'));
   ```
5. Adicione opção em `common.json → languages.<tag>` (nome no idioma nativo).
6. `npm run build` — checa que não quebrou.

## Adicionar nova chave

1. Adicione em `pt-BR/<namespace>.json` (source of truth).
2. Adicione equivalente em `en/<namespace>.json` (placeholder se sem
   tradução pronta — devops gera issue pra time de tradução).
3. Use `t('namespace.key')` no componente. Type augmentation
   (`resources.d.ts`) garante autocomplete + erro de typo.

## Pluralização

```json
{
  "empty": {
    "likes_you_cta_one": "Ver {{count}} curtida 💋",
    "likes_you_cta_other": "Ver {{count}} curtidas 💋"
  }
}
```

```tsx
t('empty.likes_you_cta', { count: likesYouCount })
```

O i18next escolhe `_one` quando `count === 1`, `_other` caso contrário. Para
russo/polonês adicione `_few`, `_many` conforme CLDR.

## Datas e horas

Não usamos um plugin de data ainda. Tempo relativo (`agora`, `5min`, `1h`)
está em `matches.json → time.*`. Para datas longas, deixar caller usar
`date.toLocaleString(i18n.language, { ... })` — funciona out-of-the-box.

## Bundle impact

Vendor chunk `vendor-i18n` carregado em paralelo ao main bundle:

```
dist/assets/vendor-i18n-*.js                 64.68 KB │ gzip: 21.44 KB
```

Inclui `i18next`, `react-i18next`, `i18next-browser-languagedetector`.
Catálogos PT-BR + EN somam ~6 KB raw / ~2 KB gzip, dentro do main.

Main bundle pós-i18n: **479 KB raw / 140 KB gzip** (vs. 466 KB / 135 KB
pré-FASE-P4 — delta de +6 KB gzip nos catálogos).

## RTL — preparado, não implementado

Setamos `<html lang="pt-BR">` estaticamente. Quando uma língua RTL (ar, he)
for adicionada:
- Detectar via `i18n.dir(locale)` que retorna `'ltr'` ou `'rtl'`.
- Setar `<html dir={dir}>` no `App.tsx` no efeito de `useTranslation`.
- Auditar CSS pra trocar `margin-left/right` por `margin-inline-start/end`.

## Comandos úteis

```bash
# Encontrar strings PT-BR remanescentes em componentes
grep -rn '[áéíóúçãõ]' frontend/src/components | grep -v node_modules

# Comparar chaves entre PT-BR e EN (deve dar diff vazio)
for f in frontend/src/i18n/locales/pt-BR/*.json; do
  base=$(basename "$f")
  diff <(jq -r 'paths|join(".")' "$f" | sort) \
       <(jq -r 'paths|join(".")' "frontend/src/i18n/locales/en/$base" | sort)
done

# Testar o app em inglês (DevTools console)
localStorage.setItem('beija_locale', 'en'); location.reload();

# Voltar
localStorage.setItem('beija_locale', 'pt-BR'); location.reload();
```
